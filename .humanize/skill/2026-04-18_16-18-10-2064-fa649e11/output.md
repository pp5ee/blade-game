CORE_RISKS:
- “成为场上最后最强者” 作为首版目标是合理的，但如果 NPC 数量、地图规模、掉落密度、出生保护没有约束，前期可能变成随机滚雪球而不是可控成长。
- 战斗“比数量和颜色，还要按公式扣刀”是核心系统，但当前没有明确结算模型；如果先做复杂公式，首版很容易陷入难平衡、难调参、难解释。
- NPC 需要同时支持拾取、追击、避战、目标切换；对浏览器单机首版来说，AI 复杂度比渲染更容易拖慢进度。
- “刀多也可自由移动、避免交战” 暗示负重或机动性变化，但规则未定义；若不处理，策略深度不足，若处理过重，会让手感变差。
- 掉落“对应数量的刀”若完全 1:1 掉落，场上实体数会膨胀，容易造成性能问题和视觉混乱。
- 视觉参考偏霓虹高密度特效，如果首版就追求图中那种光效、尾迹、环绕层次，容易挤压玩法开发时间。
- 单机网页游戏若不限定局时长，可能出现残局长时间拉扯，导致“最后最强者”体验拖沓。
- 标准交付里要求 `README.md` 没问题，但“Git commits” 在当前阶段更像交付流程要求，不属于产品需求；若计划里不区分实现与提交流程，容易混淆范围。

MISSING_REQUIREMENTS:
- 未定义技术栈；首版应直接固定为 `TypeScript + Vite + Phaser 3` 或 `TypeScript + Canvas`，否则架构无法收敛。
- 未定义输入方式；默认应为 `WASD`/方向键移动，鼠标仅用于 UI，不做主动攻击瞄准。
- 未定义地图尺寸、边界形式、出生点规则、缩圈/毒圈是否存在；这些直接决定对局节奏。
- 未定义一局人数；建议首版固定 `1 玩家 + 15~25 NPC`，避免无限扩展。
- 未定义刀的表现形式；需要明确“拾取刀”和“环绕刀”是否是同一资源对象、是否按颜色分别计数。
- 未定义角色属性；至少需要 `HP/生命是否存在`、`移速`、`拾取半径`、`刀轨道半径`、`碰撞体积`。
- 未定义战斗触发条件；应明确是“环绕刀接触敌人即结算”还是“进入近战范围时结算一次”。
- 未定义颜色公式；首版需要一个可解释的默认值，例如 `红=3, 黄=2, 蓝=1`，总战力 = 各颜色数量 × 权重，再叠加克制修正。
- 未定义颜色关系；用户只说红最强，未说是否有红黄蓝循环克制。首版建议不要做剪刀石头布循环，避免理解冲突。
- 未定义掉落规则；需要明确死亡后掉落全部刀、部分刀，还是压缩为若干拾取包。
- 未定义胜负与失败流程；玩家死亡后是立即结算、可重开，还是观战到本局结束。
- 未定义 UI 范围；至少要有刀数量、颜色分布、剩余存活数、小地图是否需要。
- 未定义音效需求；首版可默认无语音，仅做基础 SFX 钩子。
- 未定义移动端支持；若不限定，后续触控适配会大幅影响输入与 HUD 设计。
- 未定义存档与局外成长；首版应明确不做账号、不做升级、不做联网。
- 未定义可配置项；建议将 NPC 数量、掉落倍率、颜色权重、局内生成频率放入配置常量。
- 未定义 README 中的运行示例、配置示例、目录结构说明格式；计划里应把文档作为正式任务而不是收尾附带。

TECHNICAL_GAPS:
- 当前草案缺少最小可行玩法拆分，建议先做“移动 + 拾取 + 环绕显示 + 碰撞结算 + 胜负判定”，把高级 AI 和强特效后置。
- 缺少明确的实体模型；首版至少应有 `Player/NpcActor/BladePickup/OrbitBlade/MatchState/Spawner`。
- 缺少状态管理方案；需要区分“世界状态”和“表现状态”，否则后期难调试战斗公式。
- 缺少性能策略；大量刀实体不应全是独立高频对象，建议拾取物使用对象池，环绕刀只保存计数与角度槽位。
- 缺少 AI 分层；首版应做有限状态机：`Loot -> Chase -> Flee -> Recover`，用简化威胁评估，不要一开始上行为树。
- 缺少数值可调试工具；至少应预留开发面板或配置文件，用于快速调整颜色权重、逃跑阈值、刷新率。
- 缺少相机与可视范围设计；若地图大于视口，需要相机跟随和平滑边缘限制。
- 缺少碰撞结算说明；需要明确是物理引擎还是手写距离检测。首版更适合轻量距离检测。
- 缺少资源策略；当前仅有风格参考图，首版应默认用程序化几何图形和发光 shader/描边替代美术资源。
- 缺少测试策略；虽是前端游戏，也应至少定义可验证项：战力公式、掉落数量、NPC 决策阈值、胜利判定。
- 缺少项目脚手架和文档输出规划；尤其 `README.md` 必须从第一版就写清启动方式、配置项和目录结构。
- “所有提交都用 `feat:`” 不符合常规提交语义，但如果这是硬要求，可在执行计划里注明遵从；否则会影响后续维护可读性。

ALTERNATIVE_DIRECTIONS:
- 更稳妥的首版方向：做 `Canvas/Phaser` 2D 俯视角 arena，程序化霓虹图形，先放弃复杂美术，保证 60 FPS 和核心循环成立。
- 更强的玩法方向：把“颜色”做成纯权重而非克制链。默认 `红=3, 黄=2, 蓝=1`，战斗损耗按总战力差值换算，更易理解也更容易平衡。
- 更可控的掉落方向：死亡后不掉落每一把刀，而是掉落若干“刀包”，每包带颜色和数量，减少实体爆炸。
- 更易落地的 AI 方向：首版 NPC 只依据“相对战力 + 最近掉落收益 + 距离”做决策，不做群体协作、不做包夹。
- 更好的对局节奏方向：加入缓慢缩圈或危险区，防止残局无限拉扯，同时强化“成为最后最强者”的终局收束。
- 更强的可玩性方向：玩家不手动攻击，攻击完全由环绕刀触发；这样输入简单，io 风格更明确。
- 更适合首版的地图方向：单张固定 arena，无障碍或仅少量掩体，避免寻路系统复杂化。
- 更完整的实现计划方向：按三阶段推进——MVP 核心循环、可玩 AI 与数值、视觉 polish 与文档，而不是一次性铺满所有系统。
- 默认方案建议：`TypeScript + Vite + Phaser 3`，固定单机场景，20 名 NPC，8~12 分钟单局上限，程序化霓虹 UI，README 同步维护。

CANDIDATE_CRITERIA:
- 核心循环完整：玩家可移动、拾刀、增长环绕刀、与 NPC 交战、死亡掉落、继续拾取。
- 单局可闭环：从开局生成到“最后一名存活者获胜”全流程可运行，支持失败后快速重开。
- 数值可解释：颜色权重、战斗扣减、掉落数量、NPC 追逃阈值都集中配置，README 中有说明。
- 性能可接受：常规桌面浏览器下 20 名左右 NPC 和持续掉落场景保持流畅。
- AI 可玩：NPC 能表现出基本拾取、追强敌/避弱敌之外的最低限度策略变化，避免像随机游走。
- 视觉达标：整体呈现深色霓虹、俯视角、中心角色高亮、环绕彩刀的参考风格，但不过度依赖外部美术。
- 架构可扩展：实体、战斗、AI、生成器、UI 解耦，便于后续加入技能、地图、更多刀种。
- 文档完整：根目录 `README.md` 包含项目说明、环境要求、安装运行方式、配置项、代码示例、目录结构。
- 首版边界清晰：明确不做联网、不做账号、不做局外成长、不做移动端，从而保证可交付。
- 提案可执行：计划应拆成明确阶段和里程碑，而不是停留在高层玩法描述。
